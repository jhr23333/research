# Lint 知识库健康检查工作流

**适用场景**：定期（建议每月一次）扫描整个研究库，保持知识库质量。

---

## 执行步骤

### Step 1：扫描所有文件

依次读取：
1. `01_公司\` — 直接扫描目录获取覆盖公司列表（比 `_index.md` 更准确，避免失同步）
2. 每家公司的 `假设.md` — 检查假设状态
3. `02_产业链节点\` — 所有节点文件
4. `rules.md` / `false-beliefs.md`
5. `03_主题\explorations\` — 存档的讨论

### Step 2：执行五项检查

**检查A：长期僵尸假设**
找出所有状态为 🔲 待确认、且超过60天未更新的假设条目。
这些问题可能已经不重要，或者被遗忘了。

**检查B：产业链孤岛节点**
找出 `02_产业链节点\` 中没有被任何公司纪要或基本面用 `[[链接]]` 引用的文件。
孤岛节点可能是过时内容，或者需要补充引用。

**检查C：_index.md 与实际文件的一致性**
对比 `_index.md` 的公司列表与 `01_公司\` 实际存在的文件夹，找出：
- 有文件夹但未登记在index的公司
- 登记了但文件夹不存在的公司

**检查D：假设.md 与 memo.md 的矛盾**
对每家公司，检查 memo.md 中的投资逻辑是否与 假设.md 中的状态一致。
例如：memo说"看好竞争壁垒"，但对应假设状态是❌推翻。

**检查E：待核实问题积压**
统计每家公司假设.md中"待核实问题"的数量和最早提出日期，找出积压超过90天的问题。

**检查F：04_另类数据 健康度**
读取 `04_另类数据/README.md`，检查：
- 是否有数据系列缺少"关联假设"（H#为空或写"待确认"）——孤立数据，需补映射或删除
- 状态标注为"待建"超过60天的数据源——考虑降级为低优先级或移除
- data\ 目录下的 CSV 是否对应 README 中登记的数据源——多余文件需清理

### Step 3：输出检查报告

## 总体状态

|------|------|---------|------|

## B. 孤岛产业链节点
| 节点文件 | 最后更新 | 建议 |
|---------|---------|------|

## C. _index.md 不一致
{如有问题列出，否则"✅ 一致"}

| 公司 | 矛盾点 | 建议 |
|------|-------|------|

## E. 待核实问题积压（>90天）

### Step 4：收尾
询问用户：
- 是否现在处理某个具体问题
- 是否将报告存入 `03_主题/explorations/{日期}_lint报告.md`
